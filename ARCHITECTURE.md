# Smoothware AI Employee — Architecture

A CRM built to be operated jointly by human sales reps and an AI agent. This is
a **living document**: it is updated at the end of every phase so the project
stays legible as it grows.

> **Status:** Phase 0 (Foundation) + hardening + **Phase 1 (Core CRM) complete.**
> Phase 2 (Knowledge Base) not started.

---

## 1. Tech stack & why

| Layer | Choice | Rationale |
|---|---|---|
| Framework | **Laravel 13** (PHP 8.5) | Mature, relational, batteries-included. |
| Admin/UI | **Filament 5** (TALL: Tailwind, Alpine, Livewire) | A CRM/admin-panel engine — resources, tables, forms, RBAC, timelines mostly built-in. Server-rendered; no separate SPA. Its badge theming makes "AI data looks different from human data" straightforward. |
| Database | **MySQL 8** (via Docker Compose) | Fits the deeply relational model. See §7 for the RAG/vector caveat. |
| Auth & RBAC | **spatie/laravel-permission** + **Filament Shield** | Roles + per-resource policies. |
| AI layer | **Anthropic Claude API** — Opus 4.8 (reasoning), Haiku 4.5 (classification/dedup) | Used from Phase 3. RAG over the knowledge base. |
| Telephony | **Sonetel** (Dutch number) | Phase 3. REST API + call webhooks; recordings fetched into our own storage (§7). |
| Background jobs | **Laravel Queue** (database driver → Redis/Horizon later) | CSV import, embeddings, AI analysis, transcription, follow-ups all become jobs. |
| Tests | **Pest 4** | Risky/stateful logic is tested (audit log, AI + task state machines, RBAC, PII, UI render). |
| Formatting | **Laravel Pint** | |

Everything lives in **one Laravel application** (no monorepo / separate
frontend), because Filament serves the UI from within Laravel.

## 2. Phase 0 — foundational architecture (no visible UI, non-optional)

These four pillars are built **once as infrastructure** and reused by every
later phase.

### 2.1 Users, Roles & Permissions
- Roles (spatie): **`super_admin`** ("Admin" — full access), **`sales_manager`**,
  **`sales_rep`**. Seeded by `RoleSeeder`.
- Per-resource permissions are generated by Shield: **run
  `php artisan shield:generate --all --panel=admin` after adding resources** — it
  creates the permissions and assigns them to `super_admin` (that is how the
  admin bypass works; it is permission-assignment, not a gate).
- Panel access gate lives in `User::canAccessPanel()`: **active user + at least
  one role**. Archived users are excluded by the soft-delete scope.

### 2.2 Universal event log — the backbone
- Table `events`, **append-only** (`app/Models/Event.php` throws on any update
  or delete). No `updated_at`, no soft-delete.
- Columns: `entity_type`, `entity_id`, **`company_id`** (timeline anchor, §3),
  `actor_type` (`user` | `ai_agent` | `system`), `actor_id`, `action`, `payload`
  (JSON), `created_at`.
- Written **only** through `App\Services\EventLogger`, which resolves the actor:
  explicit → authenticated user → system.
- Feature models opt in with the **`LogsEvents`** trait: auto-logs create /
  update (before/after diff) / archive / restore, plus a `recordEvent('verb', …)`
  helper for domain events like `task.status_changed`.
- Powers the **Company Timeline (Phase 1)** and **Reporting (Phase 8)** — no
  per-feature ad hoc logging.
- **Reference logging for GDPR:** the log is append-only, so it can never be
  erased. Personal-data *values* are never written — a model lists PII fields in
  `$auditRedacted`; the trail records that those fields changed by name only
  (values stored as `[redacted]`). `$hidden` attributes are always redacted.

### 2.3 Soft delete everywhere
- Convention: **`archived_at`** column (not Laravel's `deleted_at`) via each
  model's `const DELETED_AT = 'archived_at'` + `SoftDeletes`.
- We archive, never hard-delete — **except** for GDPR erasure (§7).

### 2.4 AI action framework ("AI proposes → human approves → executes")
- Table `ai_actions` + `App\Services\AiActionService`. Reused by Phases 3, 4, 6.
- Lifecycle (`AiActionStatus`): `draft → approved → (applied)`, `draft →
  rejected`, or `auto_applied` (earned autonomy, still audited).
- Auditability trio on every AI record: `confidence_score`,
  `source_context_version`, `model_id`, plus `ai_run_id` and the reviewer trail.
- `apply()` runs the side effect in a transaction and records the produced record
  as the action's polymorphic **target**. Illegal transitions throw
  `InvalidAiActionTransition`.
- **Review queue (Phase 3):** a standard Filament Resource with table polling +
  database notifications — near-real-time without custom code. Only a live
  in-call transcript view needs custom Livewire + Reverb. `ai_actions(status,
  created_at)` is indexed for the poll.

## 3. Phase 1 — Core CRM

Six entities, each with soft delete, `LogsEvents`, provenance (`HasProvenance`:
`source` + `ai_action_id`), and per-model PII redaction.

- **Company** — the hub. `CompanyStatus`, owner, industry, dedup-ready
  `domain`/`phone` indexes. Detail page = tabbed relation managers (Contacts,
  Notes, Tasks, Appointments, Calls) + a read-only **Timeline**.
- **Contact** — people at a company; decision-maker flag; fully PII-redacted.
- **Note** — rich text, categorised (internal / follow-up / meeting); body redacted.
- **Task** — a real **state machine** (`TaskStatus`): `open · in_progress ·
  blocked · completed · cancelled`, reopenable. Transitions are guarded
  (`InvalidTaskTransition`), emit exactly one `task.status_changed` event, keep
  `completed_at` correct, and drive the UI action buttons. Not a boolean —
  Phase 7 automation depends on it.
- **Appointment** — Google Calendar **link-out** (`googleCalendarUrl()`); OAuth
  two-way-sync columns ready for later.
- **Call** — Phase 1 stores metadata only. Phase-3 recording/transcript/consent/
  retention columns exist now but are dormant. `transcript`/`summary` are
  **encrypted at rest**; `CallContentEraser` destroys personal content
  (recording object + transcript + numbers), keeps metadata, and audits the
  erasure — the GDPR right-to-erasure path.

**Timeline anchor:** `events.company_id` is set at write time (still append-only)
so a company's feed is one indexed query — `Event::forCompanyTimeline($id)`.
Child models derive the anchor from their `company_id`; `Company` anchors to
itself.

**Human vs. AI (principle #2):** `RecordSource` and `ActorType` implement
Filament's `HasColor`/`HasLabel`, so provenance and event actors render as
coloured badges (AI = amber). The structural half is `ai_action_id`, linking any
AI-created row to its approval record.

**Known Phase 1 follow-ups (not blockers):**
- All resource tables and the company-page relation managers now share tidy,
  consistent columns (badges, PII-safe previews); relation-manager forms fixed.
- The appointments "calendar overview" is a date-sorted list for v1; a true
  month/week calendar is a Filament plugin add (e.g. FullCalendar).
- Task block/cancel reason capture is one-click for now (reason editable on the
  record); a modal prompt can be added.

## 4. Cross-cutting conventions (apply to every new table/model)

- `id`, `created_at`, `updated_at`, `archived_at` (soft delete) unless append-only.
- `created_by` / `owner_id` on user-authored data.
- Emit an event on create/update/archive (via `LogsEvents`); list PII fields in
  `$auditRedacted`.
- AI-generated records flow through `AiActionService` and carry the auditability
  trio — never write AI data directly.

## 6. Testing
- Pest, run with `php artisan test` (42 tests). Test DB is **in-memory SQLite**
  for speed; the app targets MySQL (migrations are portable, and CI runs a
  MySQL-8 migration smoke test).
- Covered: event actor resolution + append-only enforcement, PII redaction, the
  AI-action and Task state machines (incl. illegal transitions), call content
  erasure, company-timeline anchoring, RBAC panel access, and a UI render smoke
  test across every resource page + a company detail page.
- **CI:** `.github/workflows/ci.yml` runs Pint + Pest + the MySQL migrate smoke on
  every push / PR.

## 7. Open decisions & compliance flags

- **Jurisdiction = Netherlands / EU (GDPR).**
  - **Right to erasure** — *done:* the event log never stores PII (§2.2); call
    content is destroyed by `CallContentEraser` keeping metadata. *Remaining:* a
    subject-level erasure workflow that spans a contact + their calls (Phase 3).
  - **Call recording** (Phase 3): disclosure/consent at call start, a **retention
    period**, secure storage. *Retention days + consent wording are legal
    decisions to set before go-live — not defaulted.*
  - **Outbound calling** (Phase 6): Dutch/EU telemarketing rules (do-not-call /
    right to object) — a compliance gate precedes any live queue.
  - **Sonetel**: need its API/webhook docs before Phase 3; plan is to pull
    recordings into our own object storage (MinIO/S3) so **we** control
    retention/erasure rather than the provider.
- **RAG on MySQL (Phase 2):** no native vector index; the KB is small, so
  brute-force cosine similarity in-app is sufficient. Revisit only at scale.
- **Transcript search vs. encryption (Phase 3):** encrypting the transcript
  column blocks SQL search; transcript search will use the object store / a
  dedicated index.
