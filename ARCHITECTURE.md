## 2. Phase 0 â€” foundational architecture (no visible UI, non-optional)

These four pillars are built **once as infrastructure** and reused by every
later phase.

### 2.1 Users, Roles & Permissions
- Roles (spatie): **`super_admin`** ("Admin" â€” full access, Shield's bypass
  role), **`sales_manager`**, **`sales_rep`**. Seeded by `RoleSeeder`.
- Per-resource permissions are generated by Shield (`php artisan shield:generate`)
  as Phase 1+ resources appear.
- Panel access gate lives in `User::canAccessPanel()`: **active user + at least
  one role**. Archived users are excluded by the soft-delete scope and can never
  authenticate.

### 2.2 Universal event log â€” the backbone
- Table `events`, **append-only** (`app/Models/Event.php` throws on any update
  or delete). No `updated_at`, no soft-delete.
- Columns: `entity_type`, `entity_id`, `actor_type` (`user` | `ai_agent` |
  `system`), `actor_id`, `action` (e.g. `company.created`), `payload` (JSON),
  `created_at`.
- Written **only** through `App\Services\EventLogger`, which resolves the actor:
  explicit â†’ authenticated user â†’ system.
- Feature models opt in with the **`LogsEvents`** trait
  (`app/Concerns/LogsEvents.php`): auto-logs create / update (with before/after
  diff) / archive / restore, plus a `recordEvent('verb', [...])` helper for
  domain events like `task.status_changed`.
- This single table powers the **Company Timeline (Phase 1)** and **Reporting
  (Phase 8)** â€” no per-feature ad hoc logging.

### 2.3 Soft delete everywhere
- Convention: **`archived_at`** column (not Laravel's `deleted_at`) via each
  model's `const DELETED_AT = 'archived_at'` + the `SoftDeletes` trait.
- Rationale: downstream data (AI analysis, call history) references these
  records, so we archive, never hard-delete â€” **except** for GDPR erasure (Â§6).

### 2.4 AI action framework ("AI proposes â†’ human approves â†’ executes")
- Table `ai_actions` + `App\Services\AiActionService`. Reused by Phases 3, 4, 6.
- Lifecycle (`App\Enums\AiActionStatus`): `draft â†’ approved â†’ (applied)`,
  `draft â†’ rejected`, or `auto_applied` (earned autonomy, still audited).
- Every AI-originated record carries the **auditability trio**:
  `confidence_score`, `source_context_version` (which KB + prompt-ruleset
  version), `model_id`, plus `ai_run_id` for traceability and the reviewer
  trail (`requested_by`, `reviewed_by`, `reviewed_at`, `review_notes`,
  `applied_at`).
- `apply()` runs the real side effect inside a DB transaction and records the
  produced record as the action's polymorphic **target**.
- Illegal transitions throw `InvalidAiActionTransition` â€” enforcing the
  human-in-the-loop guarantee in code, not just prompt text.
- **Review queue (answer to a Phase 3 design question):** the approve/reject
  queue is a standard Filament Resource with table polling (`->poll()`) +
  Filament database notifications â€” near-real-time without custom code. A *live
  "call in progress" transcript view* is the one piece that will need a custom
  Livewire component + Reverb WebSockets; the queue does not. The
  `ai_actions(status, created_at)` index keeps the polling query cheap.

## 3. Cross-cutting conventions (apply to every new table/model)

- `id`, `created_at`, `updated_at`, and `archived_at` (soft delete) unless the
  table is append-only.
- `created_by` / `owner_id` on user-authored data.
- Emit an event on create/update/archive (via `LogsEvents`).
- AI-generated records flow through `AiActionService` and carry the
  auditability trio â€” never write AI data directly.

## 5. Testing
- Pest, run with `php artisan test`.
- Test DB is **in-memory SQLite** (`phpunit.xml`) for speed and isolation; the

## 6. Open decisions & compliance flags

These are tracked here so they are never silently defaulted.

- **Jurisdiction = Netherlands / EU (GDPR).** The company operates a Dutch phone
  number. Consequences baked in now:
  - **Right to erasure** requires a true hard-delete path for a person's data +
    recordings, alongside normal archiving. The schema anticipates this;
    the erasure workflow itself is built when call data appears (Phase 3).
  - **Call recording** (Phase 3) needs disclosure/consent at call start, a
    defined retention period, and secure storage. *Requires an explicit product
    decision before implementation.*
  - **Outbound calling** (Phase 6) is subject to Dutch/EU telemarketing rules
    (do-not-call / right to object) â€” a compliance gate precedes any live queue.
  - Telephony provider: the Dutch number was purchased from a web provider that
    offers recording + limited settings. **We need that provider's API/webhook
    docs before Phase 3.**
- **RAG on MySQL (Phase 2).** MySQL 8 has no native vector index. The knowledge
  base is a single agency's content (hundredsâ€“low-thousands of chunks), so
  **brute-force cosine similarity computed in-app is sufficient**. Revisit
  (MySQL 9 vectors or a dedicated store) only if the KB grows large.
